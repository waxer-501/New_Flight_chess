This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Run

This is an Eclipse Java project (JDK 17+, pure standard library, no external dependencies). Output directory is `out/`, compiled classes go to `bin/`.

**Compile and run from project root:**
```bash
javac -d out src/module-info.java src/com/flightchess/app/*.java src/com/flightchess/core/*.java src/com/flightchess/net/*.java src/com/flightchess/ui/*.java
java -p out -m New_Flight_chess/com.flightchess.app.Launcher
```

**Alternative:** Double-click `Launcher.exe` in the project root — a C++ bootstrap that calls `javac`/`java` automatically. Build it from source with `launcher/build_launcher.bat` (requires MinGW-w64 g++).

**Eclipse:** Import as existing Java project. `.classpath` and `.project` are committed (in `.gitignore` though, so recommit as needed).

## Architecture

Four-layer architecture with strict separation:

```
app/  →  ui/  →  net/  →  core/
(entry)  (Swing)  (P2P TCP)  (game logic)
```

- **`core/`** — Pure game logic. `GameState` (Serializable snapshot), `RuleEngine` (stateless rule methods), `BoardConfig` (52-cell outer track constants, quarter zones, takeoff indices), `AIController` (simple greedy AI), plus data classes (`Piece`, `Player`, `PlayerColor`, `CellType`, `PlayerState`, `GamePhase`). No UI or network awareness.
- **`net/`** — P2P networking. `HostServer` runs a `ServerSocket`, adjudicates all rules, and broadcasts `GameState` snapshots via Java serialization. `ClientConnection` connects via TCP. `Message`/`MessageType` define the serializable protocol. `HostMigrationManager` and `ReconnectManager` are scaffold classes with basic strategy implementations.
- **`ui/`** — Swing GUI. `MainWindow` (nickname/join/create), `RoomWindow` (player list/ready), `GameWindow` (board + dice button), `GameBoardPanel` (custom painted board with piece click interaction). `UiController` mediates between UI and network — it creates the `HostServer`/`ClientConnection`, routes incoming messages to the right window, and forwards user actions to the server.
- **`app/`** — `Launcher.main()` calls `MainWindow.launchUi()`.

## Key design decisions

- **Host-authoritative:** All game rulings (dice rolling, move validation, capture, revival) happen on the host's `HostServer`. Clients only display state and send intent (`DICE_ROLL_REQUEST`, `MOVE_REQUEST`). Dice values are generated by the host, never by clients.
- **State sync:** The host broadcasts full `GameState` snapshots (Java `Serializable`) after every state change. There is no incremental/delta sync.
- **AI fallback:** When a player slot is empty or a player disconnects, `HostServer.runAITurnIfNeeded()` automatically runs AI turns until it reaches a connected human. AI priority: takeoff on 6 > furthest outer piece.
- **Concurrency:** `HostServer` uses `CachedThreadPool` for client handlers. Move requests are serialized with `moveRequestLock`. `GameBoardPanel.moveRequestPending` prevents double-clicks.
- **Board layout:** 52-cell outer track, 13 per side. `BoardConfig` maps colors to quarter sections: GREEN [0,12] bottom, YELLOW [13,25] left, BLUE [26,38] top, RED [39,51] right. Takeoff cells use negative indices (-1 to -4). Cell coloring cycles GREEN→RED→BLUE→YELLOW every 4 cells (`outerIndex % 4`).

## PRD vs implementation gap

The PRD (`docs/PRD.md`) specifies many advanced rules that are **not yet implemented** in `RuleEngine`:
- Sudden death mode (free movement, dual dice, center path traversal)
- Air routes (航道) and colored-cell auto-jump chains
- Double zones from dead players
- Stacked piece capture rules
- Full host migration on disconnect (scaffold exists)

The current `RuleEngine` implements: basic takeoff (6 → TAKEOFF cell → outer track), forward movement, capture (move to WAITING_AREA), consecutive-6 revival, and extra turn on 6/capture. Cell-color auto-advance is limited to +4 on landing on own-color cell.
